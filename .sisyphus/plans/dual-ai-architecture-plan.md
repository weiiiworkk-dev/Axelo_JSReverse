# 双AI架构系统重构计划

## 核心理念

**消除站点微调，拥抱通用能力**

> 系统应具备识别任何电商/网站签名算法的通用能力，而非针对特定站点硬编码规则。

---

## 一、当前问题诊断

### 1.1 现有架构问题

| 问题 | 表现 | 根本原因 |
|------|------|----------|
| Intent分类依赖硬编码正则 | "亚马逊 苹果"无法识别为逆向任务 | 关键词驱动的规则系统 |
| 工具选择逻辑脆弱 | confidence<0.6时丢弃AI选择 | 阈值驱动的fallback逻辑 |
| 签名提取无上下文 | 每次执行独立，无历史学习 | 工具间无状态传递 |
| 验证失败即终止 | 遇到反爬就停止 | 缺少重试/降级策略 |

### 1.2 用户明确约束

- ❌ **禁止**: 对任何站点进行微调
- ❌ **禁止**: 站点特定规则/配置
- ✅ **允许**: 工具能力增强
- ✅ **允许**: Prompt工程优化

## 二、双AI架构设计

### 2.1 架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                      用户输入层                                 │
│  "逆向亚马逊，爬取苹果产品"                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│              AI-1: Planner (规划器)                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐            │
│  │ Intent分析   │→ │ 工具规划    │→ │ 执行计划    │            │
│  │ (理解目标)   │  │ (选择工具)  │  │ (参数配置)  │            │
│  └─────────────┘  └─────────────┘  └─────────────┘            │
│         ↓                ↓                ↓                     │
│  输出: {intent, tools[], params{}, reasoning}                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│              AI-2: Executor (执行器)                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐            │
│  │ 工具调度     │→ │ 状态管理    │→ │ 异常处理    │            │
│  │ (按序执行)  │  │ (上下文)    │  │ (重试/降级) │            │
│  └─────────────┘  └─────────────┘  └─────────────┘            │
│         ↓                ↓                ↓                     │
│  输出: {results[], artifacts[], execution_log}                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      输出层                                     │
│  生成的爬虫代码 + 验证报告 + 执行日志                            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 AI-1 Planner 职责

**核心任务**: 理解用户意图，生成执行计划

| 功能模块 | 输入 | 输出 | 技术实现 |
|----------|------|------|----------|
| **意图理解** | 用户输入+URL | 意图类型、目标数据、复杂度 | Prompt工程 |
| **工具规划** | 意图+站点特征 | 工具序列+执行顺序 | Prompt工程 |
| **参数配置** | 工具序列+上下文 | 每个工具的输入参数 | Prompt工程 |
| **假设生成** | 分析结果 | 签名算法假设+置信度 | LLM推理 |

**关键设计原则**:

1. **零硬编码关键词**: Intent分类完全由LLM通过Prompt判断
2. **动态工具选择**: 基于站点特征而非预设规则
3. **参数自动推断**: 工具参数由LLM根据上下文生成

### 2.3 AI-2 Executor 职责

**核心任务**: 执行计划，管理工作流

| 功能模块 | 输入 | 输出 | 技术实现 |
|----------|------|------|----------|
| **工具调度** | 工具序列+参数 | 工具执行结果 | 已有executor |
| **上下文传递** | 前置工具输出 | 后续工具输入 | 状态管理 |
| **异常处理** | 执行失败 | 重试/降级/报告 | 规则引擎 |
| **结果聚合** | 全部工具输出 | 最终产物+报告 | 数据整合 |

## 三、Prompt工程方案

### 3.1 Planner Prompt 设计

```python
PLANNER_SYSTEM_PROMPT = """你是一个专业的AI爬虫系统架构师。

## 你的角色
你负责分析用户请求，制定逆向工程和爬虫执行计划。你不需要执行代码，只负责规划。

## 核心能力
1. 意图理解：准确识别用户想要的結果
2. 工具规划：选择最适合的工具组合
3. 参数生成：为每个工具生成正确的输入参数

## 意图类型定义
- reverse_engineer: 逆向签名算法 (如"逆向亚马逊签名", "提取淘宝加密")
- crawl: 简单爬取 (如"爬取商品列表", "下载图片")
- analyze: 仅分析JS (如"检查网站用了什么加密")
- bypass: 绕过反爬 (如"绕过验证码", "突破IP限制")

## 站点复杂度判断
根据以下特征判断:
- 简单: 静态HTML，无签名
- 中等: 基础JS，简单签名
- 复杂: 混淆JS，自定义签名
- 极复杂: WASM，强反爬

## 工具能力库
- browser: Playwright浏览器，获取页面+动态执行
- fetch: HTTP请求，获取HTML
- fetch_js_bundles: 提取页面引用的JS文件
- static: 静态JS代码分析，提取候选签名
- crypto: 加密算法识别
- signature_extractor: 从JS提取密钥/算法
- ai_analyze: AI分析，生成假设
- codegen: 生成爬虫代码
- verify: 验证代码可用性
- trace: 调用链追踪
- deobfuscate: 反混淆

## 输出格式
请返回以下JSON格式:

{
  "intent": {
    "type": "reverse_engineer|crawl|analyze|bypass",
    "confidence": 0.0-1.0,
    "reasoning": "你的分析过程",
    "complexity": "simple|medium|complex|very_complex"
  },
  "plan": {
    "tools": ["tool1", "tool2", ...],
    "tool_params": {
      "tool1": {"param1": "value1"},
      "tool2": {"param2": "value2"}
    },
    "reasoning": "为什么选择这些工具"
  }
}

## 重要规则
1. 不要假设任何站点特定信息
2. 基于通用特征判断复杂度
3. 如果不确定，选择更强大的工具组合
4. 始终包含 verify 工具作为最后一步"""

PLANNER_USER_PROMPT = """用户请求: {user_input}
目标URL: {url}

请分析并生成执行计划。"""
```

## 五、关键技术决策

### 5.1 LLM选择

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| Planner (意图分析) | DeepSeek-Chat | 快速、便宜、准确 |
| Planner (工具规划) | DeepSeek-Chat | 需结构化输出 |
| 执行中决策 | 规则引擎 | 无需LLM，规则驱动 |

### 5.2 状态管理

```python
@dataclass
class ExecutionState:
    plan: Plan  # Planner生成的执行计划
    context: dict  # 工具间传递的上下文
    results: dict[str, ToolResult]  # 每个工具的执行结果
    current_tool: str  # 当前执行的工具
    history: list[ExecutionStep]  # 执行历史
```

### 5.3 错误处理策略

| 错误类型 | 处理策略 |
|----------|----------|
| 工具执行失败 | 重试1-2次，失败则跳过并记录 |
| LLM解析失败 | 使用默认计划，标记为"未确认" |
| 验证失败 | 记录详细错误，返回给用户 |
| 超时 | 跳过工具，继续执行后续 |
