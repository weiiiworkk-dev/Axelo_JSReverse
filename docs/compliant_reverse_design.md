# 符合风控规格的逆向方式

## 1. 设计原则

这里的“符合风控规格”不是绕过风控，而是：

- 只在授权、允许、可审计的范围内工作
- 一旦命中挑战页、验证码、惩罚页或账号/设备校验，立即停止自动对抗
- 把系统设计成“检测与诊断工具”，而不是“规避与对抗工具”

一句话原则：

- 可识别
- 可暂停
- 可审计
- 可切换到官方通道

## 2. 从这次日志看到的事实

当前 `4f51e134` 会话里，已经出现明确的风控命中信号：

- `FAIL_SYS_USER_VALIDATE`
- `RGV587_ERROR`
- `punish?x5secdata=...`

这说明当前回放流量已经被服务端识别为挑战流量，不应继续自动重试。

## 3. 合规逆向架构

### A. 目标定义层

输入必须包含：

- 目标 URL
- 逆向目标
- 目标对象提示 `target_hint`
- 数据用途
- 授权来源

系统要求：

- 若无 `target_hint`，仅允许做发现，不允许直接产出自动抓取器
- 若无授权说明，只允许做静态分析和人工审阅，不做实时回放

### B. 发现层

允许做的事：

- 浏览器打开页面
- 记录网络请求
- 识别 JS 资源
- 记录响应类型和挑战信号

禁止做的事：

- 自动重复挑战请求
- 自动轮换身份去对抗风控
- 自动伪造设备或环境信号

### C. 特征提取层

职责：

- 提取签名相关字段
- 识别请求家族
- 判断是否存在风控依赖

输出必须区分两类字段：

- 业务字段
  - 路径
  - 查询参数
  - body
  - 普通 header
- 风控字段
  - 设备指纹
  - 行为轨迹
  - 验证 token
  - 挑战参数

如果请求依赖风控字段，系统必须标记为：

- `requires_authorized_runtime`

而不是自动继续生成全自动 crawler。

### D. 回放层

回放只能分 3 档：

#### 1. 安全回放

适用：

- 公共接口
- 不依赖登录
- 不依赖挑战参数
- 不依赖设备指纹

动作：

- 单次回放
- 低频
- 仅用于验证字段完整性

#### 2. 授权运行时回放

适用：

- 需要登录态
- 需要用户明确授权
- 依赖浏览器上下文

动作：

- 仅在受控浏览器上下文里执行
- 不脱离原始会话环境
- 不导出可泛化的绕过逻辑

#### 3. 停止回放

适用：

- 命中挑战页
- 返回惩罚页
- 命中验证码
- 返回用户验证失败

动作：

- 立刻停止自动重试
- 生成诊断报告
- 建议改走官方或授权接口

## 4. 风控识别规范

系统应把以下信号视为强风控命中：

- HTTP 401 / 403
- `FAIL_SYS_USER_VALIDATE`
- `RGV587_ERROR`
- `/_____tmd_____/punish`
- `x5secdata`
- challenge 页面标题或验证码组件

处理规则：

- 单次命中后立即停止自动重试
- 将状态写入 `run_report.json`
- 在 CLI 里输出“风控命中摘要”
- 不再触发高成本 AI 修复链路

## 5. 符合风控规格的产物边界

允许输出：

- 请求家族识别结果
- 签名步骤说明
- 字段清单
- 需要哪些授权上下文
- 官方接口/合规通道建议
- 只用于受控环境的验证脚本

不应输出：

- 用于泛化规避风控的自动化脚本
- 自动伪造设备/行为指纹的代码
- 面向未知第三方目标的大规模回放器

## 6. CLI 应该如何体现“合规”

CLI 不应该只是“能跑”，还要能约束。

### 必须新增的约束

1. 增加 `用途说明`

- 调试
- 研究
- 自有系统
- 授权合作方

2. 增加 `授权状态`

- 已授权
- 待确认
- 未授权

3. 增加 `回放模式`

- `discover_only`
- `authorized_replay`
- `official_api_only`

### 必须新增的拦截

- `未授权 + 命中风控` 时禁止继续自动化
- `首页 + 无 target_hint` 时禁止直接 `auto`
- `检测到挑战页` 时禁止重试 codegen 和 replay

## 7. 推荐的实现方式

### P0

- 风控挑战检测器
- 早停机制
- CLI 风控摘要页
- `target_hint` 强制化

### P1

- `SignatureFamilyDetector`
- `requires_authorized_runtime` 标记
- 官方 API / 导出 / 合作接口适配器

### P2

- 授权运行时容器
- 审计日志
- 人工审批点
- 合规模式下的模板 codegen

## 8. 合规模式下的“逆向”

如果一定要做逆向，建议限定成下面这种形式：

- 目标是理解协议，不是绕过防护
- 目标是生成 `SignatureSpec`，不是生成泛用对抗脚本
- 目标是判断“哪些字段是业务协议，哪些字段是风控协议”
- 目标是帮助迁移到官方通道、受控运行时或人工审核流程

## 9. 最终建议

Axelo 的“合规模式”应该变成：

- 自动化发现
- 风控识别
- 授权判定
- 结构化诊断
- 官方通道推荐
- 受控回放

而不是：

- 自动抓取
- 自动重试
- 自动对抗
- 自动绕过

这样做的好处是：

- 法务风险更低
- 平台风险更低
- 成本更低
- 结果更稳定
- 产物更容易沉淀为长期可维护的系统
